# Анализ проекта SyMo и предложения по развитию

Дата анализа: 2026-04-08

## 1) Что уже реализовано

### Продуктовые возможности
- Лёгкий Linux tray-монитор (GTK3/AppIndicator) с ключевыми метриками: CPU, RAM, swap, disk, сеть, uptime, активность клавиатуры и мыши.
- Окна графиков по метрикам с интерактивом (zoom/pan/hover-подсказки).
- Управление питанием: shutdown/reboot/lock + отложенный таймер.
- Интеграции уведомлений: Telegram (включая `/status` и `/screenshot`) и Discord webhook.
- Настраиваемое меню в трее: включение/выключение пунктов + сортировка.
- Мультиязычность (8 языков).

### Техническое состояние
- Кодовая база разделена по модулям (`app_core`, `notifications`, `tests`) и выглядит поддерживаемо.
- Есть тесты на устойчивость конфигов, локализацию ключей, доставку уведомлений, порядок меню и “hardening”.
- Реализована retry/backoff-логика для Telegram/Discord и очередь отправки уведомлений в фоне.
- Есть скрипты сборки/упаковки и деинсталляции.

Итог: проект уже в рабочем состоянии как desktop utility c хорошей функциональной базой.

---

## 2) Ключевые сильные стороны

1. **Хорошая прикладная ценность «из коробки»** — мониторинг + быстрые действия + удалённые уведомления.
2. **Неплохая устойчивость на сетевом контуре** — retry/backoff и ограничение длины сообщений.
3. **Продуктовая гибкость** — настройки видимости пунктов меню, история графиков, языки.
4. **Наличие автотестов** — для такого типа desktop-проекта это уже плюс к качеству.

---

## 3) Что ограничивает рост сейчас

1. **Нет формализованной дорожной карты и архитектурных ADR**
   - Порог входа для контрибьюторов выше, чем мог бы быть.

2. **Мало наблюдаемости для эксплуатации**
   - Логи есть, но нет единого раздела health/diagnostics (например, состояние интеграций, частота ошибок, последняя успешная отправка).

3. **Ограниченный набор интеграций уведомлений**
   - Только Telegram/Discord, отсутствуют Slack, email, ntfy, Matrix, Gotify.

4. **Нет “правил” уведомлений (alerting engine)**
   - Сейчас упор на периодические сообщения и ручные команды; не хватает событийных алертов (CPU>90% 5 минут, диск<10%, сеть down и т.п.).

5. **Не хватает user-facing аналитики по историческим данным**
   - История есть в памяти для графиков, но нет долговременного хранения и простых инсайтов (пики, средние, тренды за день/неделю).

6. **Нет CI/CD уровня репозитория**
   - Тесты запускаются локально, но отсутствуют явные workflow для автоматической проверки PR/релизов.

7. **Ограниченное покрытие UX и platform-specific сценариев**
   - Для Linux desktop важно тестировать различия X11/Wayland/DE; это можно усилить хотя бы smoke-check списком и документированными known issues.

## 4) Что добавить в проект для развития (приоритетно)

## P0 (ближайший цикл, максимальный эффект)

### A. Система правил уведомлений (Alert Rules)
**Что добавить**
- Конфигурируемые правила в UI:
  - CPU/RAM/swap/disk/network thresholds;
  - длительность условия (например, 3–5 минут);
  - cooldown (чтобы не спамить);
  - каналы доставки (Telegram/Discord).

**Почему важно**
- Это переводит SyMo из “монитора по запросу” в “проактивный мониторинг”.

### B. Диагностический экран/раздел
**Что добавить**
- Статус интеграций: enabled, валидность токена/webhook, last success/last error, время последней отправки.
- Кнопка “Собрать диагностический отчёт” (без секретов), который удобно прикладывать в issue.

**Почему важно**
- Снижает время разборов пользовательских проблем.

### C. CI-пайплайн (GitHub Actions)
**Что добавить**
- Автозапуск `pytest -q` на push/PR.
- Проверки форматирования/линтинга (`ruff`, `black --check` или эквивалент).
- Базовую проверку сборки (smoke build).

**Почему важно**
- Стабилизирует развитие при росте кода и контрибьюторов.

---

## P1 (следующий этап)

### D. Плагинная модель каналов уведомлений
**Что добавить**
- Абстракция `Notifier` + registry провайдеров.
- Поставляемые плагины: Telegram, Discord (существующие), затем Slack/ntfy/Gotify.

**Почему важно**
- Упрощает масштабирование интеграций без усложнения основной логики.

### E. Историческое хранилище метрик
**Что добавить**
- Лёгкое локальное хранилище (SQLite) с retention policy (например, 7/30 дней).
- UI-режим “за день/неделю” с агрегатами min/max/avg.

**Почему важно**
- Появляется аналитическая ценность и полезность “после инцидента”.

### F. Профили конфигурации
**Что добавить**
- Профили “Ноутбук / Рабочая станция / Сервер desktop” с разными предустановками частоты опроса, графиков, правил.

**Почему важно**
- Быстрый onboarding без ручного тюнинга десятков параметров.

## P2 (среднесрочно)

### G. Экспорт/импорт настроек
- Экспорт в JSON (без секретов или с отдельным флагом).
- Импорт и валидация версии схемы настроек.

### H. Расширение Telegram-команд
- `/health`, `/alerts`, `/mute <minutes>`, `/unmute`, `/netcheck`, `/logs` (ограниченно и безопасно).

### I. Улучшение безопасности секретов
- Поддержка хранения секретов в keyring (если доступен), а не только в файлах.
- Явный security-раздел в README.

## 7) Метрики успеха развития

Рекомендуемые KPI после внедрения roadmap:
- Доля активных пользователей с включёнными alert rules.
- Среднее число ложных уведомлений на пользователя в неделю.
- Время до обнаружения инцидента (MTTD) по событиям CPU/disk/network.
- Процент успешной доставки уведомлений по каналам.
- Стабильность релизов: % зеленых CI прогонов и число регрессий.

## 8) Какой ещё функционал стоит добавить (дополнительно)

Ниже — дополнительные идеи, которые можно включать после базового roadmap.

### Пользовательские сценарии и UX
- **Профили режима работы**: «Работа», «Игра», «Энергосбережение» (разные интервалы опроса, наборы метрик, правила уведомлений).
- **Quiet Hours / DND**: временные окна, когда уведомления приглушаются или отправляются только критические.
- **Быстрые пресеты панели**: компактный/детальный вид для трея.
- **Мастер первого запуска**: проверка зависимостей, первичная настройка Telegram/Discord, тест отправки.

### Расширение мониторинга
- **Мониторинг GPU**: загрузка, температура, VRAM (где доступно).
- **Мониторинг батареи/питания** (для ноутбуков): износ батареи, прогноз времени, события «низкий заряд».
- **Процессы-топы**: топ-5 по CPU/RAM/IO в отдельном окне.
- **Сетевые endpoints checks**: health-check до выбранных URL/хостов + история задержек.

### Оповещения и автоматизация
- **Комбинированные правила**: «CPU > 85% И temp > 80°C 5 минут».
- **Action hooks**: пользовательские команды при срабатывании правила (с подтверждением и безопасным allowlist).
- **Эскалация каналов**: сначала Discord, затем Telegram при отсутствии подтверждения.
- **Mute/Snooze per alert**: временное отключение отдельных правил.

### Управление данными
- **Экспорт истории графиков** в CSV/JSON для внешнего анализа.
- **Retention-политики** по типам данных (сырые/агрегированные).
- **Бэкап/восстановление конфигов** с версионированием схемы.

### Безопасность
- **Секреты в системном keyring** + опционально env-overrides.
- **Редактирование чувствительных логов**: автоматическая маскировка токенов/chat_id/webhook.
- **Permission-check экран**: что доступно в текущей сессии (Wayland/X11, screenshot backends, AppIndicator).

### Экосистема
- **Новые каналы уведомлений**: Slack, Matrix, ntfy, Gotify, email.
- **Плагин API** для сторонних модулей метрик/каналов.
- **Импорт Prometheus node_exporter textfile** (экспорт метрик SyMo наружу).

### Качество и DevEx
- **Crash reporting (opt-in)** с локальным сохранением отчёта.
- **Диагностический self-check** перед релизом (скрипт smoke-тестов окружения).
- **Стабильные релизные артефакты**: версионирование, changelog, checksums, подписи.

### Рекомендуемый следующий инкремент после P0
Если P0 закрыт, оптимальный следующий шаг —
1) SQLite-история + агрегаты,
2) GPU/батарея метрики,
3) один новый канал уведомлений (например, ntfy),
4) quiet hours + snooze.
