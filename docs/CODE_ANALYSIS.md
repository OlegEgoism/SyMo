# SyMo: глубокий анализ кода

## 1) Обзор архитектуры

Проект построен вокруг монолитного рантайм-класса `SystemTrayApp`, который:
- инициализирует GTK/AppIndicator,
- поднимает слушатели глобального ввода (`pynput`),
- запускает периодическое обновление метрик,
- управляет окнами графиков,
- делегирует power-действия в `PowerControl`,
- делегирует уведомления в `TelegramNotifier` / `DiscordNotifier`.

Точка входа (`app.py`) остаётся «тонкой» и просто запускает `SystemTrayApp` из `app_core/app.py`.

### Ключевые слои

- **UI/Orchestration:** `app_core/app.py`, `app_core/dialogs.py`, `app_core/power_control.py`.
- **Domain/services:** `app_core/system_usage.py`, `app_core/click_tracker.py`, `app_core/localization.py`.
- **Integrations:** `notifications/telegram.py`, `notifications/discord.py`.
- **Configuration/constants:** `app_core/constants.py`.
- **Reliability tests:** `tests/`.

## 2) Поток данных и жизненный цикл

1. При старте читаются настройки и выставляется язык интерфейса.
2. Создаётся системный индикатор и меню с пунктами метрик и действий.
3. Инициализируются глобальные слушатели клавиатуры/мыши.
4. Таймер GTK раз в `TIME_UPDATE_SEC` сек. обновляет метрики.
5. В том же цикле (при включении) триггерятся Telegram/Discord уведомления по интервалу.
6. При завершении корректно останавливаются слушатели и бот-поток Telegram.

## 3) Что реализовано качественно

- **Thread-safety для счётчиков кликов:** отдельный lock в `click_tracker.py`.
- **Fail-safe по метрикам:** `SystemUsage.get_cpu_temp()` имеет fallback-логику по разным сенсорам и не роняет приложение при исключениях.
- **Нормализация интервалов уведомлений:** в обоих notifiers (`10..86400` секунд).
- **Failover команд питания:** `loginctl` → `systemctl`; для lock — несколько команд-кандидатов.
- **Файловые права на секреты:** для конфигов нотификаторов выставляется `0600`.
- **Покрытие edge cases тестами:** проверяются fallback-и интервалов, `force=True` отправки, устойчивость к отсутствию окружения GI.

## 4) Технические риски и узкие места

### 4.1 Перегруженный `SystemTrayApp`
`app_core/app.py` содержит много ответственности сразу: orchestration, сбор метрик, рендер нескольких графиков, i18n-меню, уведомления, обработчики событий. Это повышает стоимость изменений и риск регрессий.

**Рекомендация:** выделить минимум 3 сервиса:
- `MetricsService` (polling + snapshot),
- `NotificationService` (rate-limit + форматирование + fan-out),
- `GraphPresenter` (история + draw callbacks).

### 4.2 Дублирование логики графиков
Есть повторяемый паттерн `append/show/on_destroy/refresh/draw` для CPU/RAM/SWAP/Disk/Net/Keyboard/Mouse.

**Рекомендация:** ввести абстракцию `MetricGraphWindow` с параметрами:
- ключ метрики,
- formatter,
- цвета,
- maxlen/history source.

### 4.3 Смешение UI и бизнес-логики
В `dialogs.py` тест отправки уведомлений одновременно валидирует ввод, пишет конфиг и шлёт сетью. Это затрудняет unit-тестирование UI отдельно от интеграций.

**Рекомендация:** перенести проверку/сохранение/тестовую отправку в сервисный слой, а диалог оставить «тонким».

### 4.4 Наблюдаемость и логирование
Логирование в основном через `print()`. Есть ротация лога, но нет уровней, контекста и единой схемы логов.

**Рекомендация:** перейти на `logging` (structured-ish), оставить fallback в консоль только при раннем старте.

### 4.5 Устойчивость к внешним API
Telegram polling уже имеет backoff, но стратегия достаточно базовая.

**Рекомендация:**
- jitter для backoff,
- явная типизация/валидация входящих update payload,
- единая политика retry для Telegram/Discord.

## 5) Анализ тестового покрытия

Тесты в текущем виде подтверждают важные «контуры надёжности»:
- согласованность языков и переводов,
- корректная нормализация интервалов,
- возможность принудительной отправки (`force=True`) при `enabled=False`,
- наличие backoff-логики в Telegram polling,
- корректность uninstall-сценария для вариантов desktop-файлов.

### Что стоит добавить

1. **Unit-тесты на `SystemUsage.get_network_speed()`** (нулевой/малый elapsed, non-negative delta).
2. **Тесты на `PowerControl._run_command()`** с моками возвратов/ошибок.
3. **Тесты на восстановление `settings` при повреждённом JSON** (как уже сделано для notifier-конфигов).
4. **Snapshot tests форматирования текста метрик** (готовая строка в меню/нотификациях).

## 6) Практический roadmap (без ломки API)

### Фаза 1 — минимально рискованные улучшения
- Вынести форматирование уведомлений в отдельные функции.
- Вынести повторяющуюся логику графиков в helper-класс.
- Унифицировать обработку ошибок `requests` (общий wrapper).

### Фаза 2 — структурная декомпозиция
- Ввести сервисы `MetricsService` / `NotificationService`.
- Упростить `SystemTrayApp` до orchestration + UI wiring.
- Поднять покрытие юнит-тестами на сервисный слой.

### Фаза 3 — операционная надёжность
- Перейти на стандартный `logging` и расширить ротацию.
- Добавить диагностический режим (verbose) для проблем окружения GTK/GI.
- Добавить health-check команду в Telegram bot (`/health`) c краткой диагностикой окружения.

## 7) Краткий итог

Проект уже имеет сильный фундамент по функциональности и базовой отказоустойчивости. Основной потенциал улучшения — **снижение связности в `SystemTrayApp` и устранение дублирования графического слоя**. Это даст более быстрые изменения, легче тестируемый код и проще поддержку новых метрик/каналов уведомлений.
