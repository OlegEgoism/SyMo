# SyMo — глубокий анализ кода и структуры проекта

## 1. Назначение проекта

SyMo — это Linux-приложение в системном трее (GTK + AppIndicator/Ayatana), которое:

- показывает системные метрики в реальном времени (CPU, RAM, SWAP, диск, сеть, uptime);
- считает глобальные клики клавиатуры/мыши через `pynput`;
- выполняет power-действия (выключение/перезагрузка/блокировка), включая отложенный запуск;
- отправляет периодические уведомления в Telegram и Discord;
- поддерживает локализацию интерфейса.

Точка входа: `app.py`.

## 3. Архитектура и поток выполнения

## 3.1 Инициализация (`SystemTrayApp.__init__`)

1. Загружаются настройки (`~/.symo_settings.json`) c fallback-дефолтами.
2. Определяется и активируется язык интерфейса.
3. Создаётся системный индикатор и меню GTK.
4. Инициализируются listener-ы `pynput` для клавиатуры/мыши.
5. Создаются Telegram/Discord notifier-ы, при необходимости запускается Telegram bot thread.
6. При включённом логировании создаётся файл лога.

## 3.2 Основной цикл

- `run()` запускает `GLib.timeout_add_seconds(TIME_UPDATE_SEC, update_info)`.
- `update_info()` каждую секунду:
  - читает метрики (`SystemUsage`);
  - обновляет пункты меню и label в трее;
  - при наступлении интервала отправляет Telegram/Discord уведомления в background thread;
  - пишет строку в лог с ротацией.

## 3.3 Слои ответственности

- **UI/Orchestration:** `app.py`, `dialogs.py`, `power_control.py`
- **Domain/metrics:** `system_usage.py`, `click_tracker.py`
- **Integrations:** `notifications/telegram.py`, `notifications/discord.py`
- **I18n:** `language.py`, `localization.py`
- **Infrastructure:** `constants.py`, `logging_utils.py`, `build.sh`

## 4. Форматы данных и хранение

Файлы в `$HOME`:

- `~/.symo_settings.json` — UI-настройки и поведение.
- `~/.symo_telegram.json` — токен, chat id, интервал, enabled.
- `~/.symo_discord.json` — webhook, интервал, enabled.
- `~/.symo_log.txt` — runtime-метрики по таймеру.

Замечание по безопасности: для Telegram/Discord конфигов выставляется `chmod 600`.

## 5. Сильные стороны текущей реализации

- Простой и понятный модульный разрез.
- Потокобезопасный трекинг кликов (`threading.Lock`).

- Защита от отсутствия AppIndicator за счёт fallback на Ayatana.
- Базовое тестовое покрытие критичных утилитарных модулей.

## 6. Выявленные проблемы и что исправить

Ниже — приоритетный backlog по улучшениям.

## P0 (исправить в первую очередь)

1. **Риск разрушения пользовательского состояния при тестах из-за глобальных путей конфигов.**
   - Сейчас тесты monkeypatch-ят константы в импортированном модуле (например, `DISCORD_CONFIG_FILE`), но общий подход не систематизирован.
   - **Что сделать:** перейти на dependency injection путей через параметры конструктора notifier-ов/настроек, либо централизованный слой конфигурации.

2. **Слабая валидация и silent-fail при JSON-конфигах.**
   - В ряде мест исключения заглушаются `except Exception: pass`, что маскирует повреждённые конфиги.
   - **Что сделать:** логировать контекст ошибки (файл, причина), откатываться на дефолт и уведомлять пользователя в UI (мягко).

## P1 (важно)

3. **Несоответствие заявленных языков README и фактической поддержки.**
   - В README упомянут арабский, но в `SUPPORTED_LANGS` его нет.
   - **Что сделать:** либо удалить Arabic из README, либо добавить `ar` в `SUPPORTED_LANGS` и словарь `language.py`.

4. **Дубли ключей локализации в русской секции (`language.py`).**
   - Ключи `bot_shutdown_message`, `bot_reboot_message`, `bot_help_message` объявлены дважды; фактически используются последние значения.
   - **Что сделать:** удалить дубли, добавить тест на уникальность ключей в исходнике.

5. **Telegram bot polling без backoff/circuit-breaker.**
   - При нестабильной сети цикл может слишком агрессивно продолжаться.
   - **Что сделать:** добавить экспоненциальный backoff при `RequestException`, ограничение на частоту ошибок.

6. **Power actions через `os.system` без централизованной проверки/логирования результата.**
   - Сложно отлаживать причину неуспеха на разных DE/дистрибутивах.
   - **Что сделать:** перейти на `subprocess.run(..., check=False, capture_output=True)`, писать stderr в лог.

## P2 (желательно)

7. **Смешение языка кода и сообщений.**
   - Сообщения для консоли/ошибок в основном на русском, что усложняет сопровождение международной командой.
   - **Что сделать:** привести runtime/log-сообщения к одному стандарту (например, English), а UI оставить локализуемым.

8. **Неполное тестовое покрытие.**
   - Нет юнит-тестов для `system_usage.py`, `power_control.py`, `localization.py`, `logging_utils.py`, Telegram части.
   - **Что сделать:** покрыть критические сценарии + негативные кейсы (битый JSON, таймауты API, отсутствие sensors).

9. **`uninstall-symo.sh` и build naming mismatch.**
   - Скрипт удаления ищет `symo.desktop`, а build создаёт `${APP_NAME}.desktop` (`SyMo.desktop`).
   - **Что сделать:** унифицировать имена (lowercase/uppercase) в build и uninstall.

10. **Обновление меню пересозданием (`create_menu`) может приводить к накоплению ссылок и сложной поддержке.**
    - **Что сделать:** рефакторинг в stateful update без полного recreate, либо явное разрушение старого меню и повторное связывание.

Ниже — максимально прикладной список улучшений с ожидаемым эффектом.

1. **Синхронизировать языки в README и коде**
   - Сейчас README и `SUPPORTED_LANGS` расходятся по Arabic.
   - Эффект: убирает пользовательскую путаницу и несоответствие документации.

2. **Удалить дубли ключей в `language.py`**
   - Убрать повторные `bot_*` ключи и добавить тест на дубликаты.

3. **Улучшить диагностику битых JSON-конфигов**
   - В местах загрузки конфигов не глушить исключения, а писать понятный warning.

   - Привести к одному стилю (`SyMo.desktop` или `symo.desktop`) во всех скриптах.

### 10.2 Улучшения надёжности (2–4 дня)

   - Логировать `returncode`, `stdout/stderr` и команду.

6. **Добавить backoff в Telegram polling**
   - Экспоненциальная задержка при сетевых ошибках + ограничение max delay.
   - Эффект: меньше нагрузка на сеть/API, стабильнее поведение при деградациях.

7. **Разделить конфиг-слой и UI-слой**
   - Вынести чтение/запись настроек в отдельный сервис.
   - Эффект: проще тестировать и сопровождать, меньше связность `app.py`.

8. **Расширить тестовое покрытие**
   - Добавить тесты на `localization.py`, `logging_utils.py`, `system_usage.py`, базовые кейсы `power_control.py`.

9. **Ввести quality-gates в CI**
   - `pytest`, линтер (`ruff`/`flake8`), форматтер (`black`), базовая типизация (`mypy`/`pyright`).
   - Эффект: единый стандарт качества и меньше дефектов при merge.

10. **Добавить smoke-check для runtime-зависимостей GTK/AppIndicator**
   - Скрипт/док-проверка окружения перед запуском.
   - Эффект: меньше ложных баг-репортов «не стартует приложение».

## 11. Проблемные места (короткая карта рисков)

- **Конфигурация и отказоустойчивость:** неявные ошибки при чтении конфигов, слабая диагностика.
- **Интеграции:** Telegram polling без backoff; сетевые проблемы могут приводить к нестабильному циклу.
- **Системные действия:** power-команды через `os.system` затрудняют отладку и аудит выполнения.
- **Локализация:** дубли ключей и частичная несогласованность заявленных языков.
- **Скрипты поставки:** разный нейминг desktop-файлов в build/uninstall.
- **Тесты:** покрытие есть, но фрагментарное для инфраструктурных и интеграционных модулей.

## 12. Рекомендуемая последовательность внедрения

1. **Документационная консистентность + локализация** (быстро и безопасно).
2. **Надёжность runtime** (backoff, `subprocess.run`, улучшенный error reporting).
3. **Тесты и CI** (закрепить изменения quality-gates).
4. **Модульный рефакторинг** (конфиг-сервис, снижение связности `app.py`).

Такой порядок даёт лучший баланс между скоростью и снижением эксплуатационных рисков.
